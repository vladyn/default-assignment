Assignment documentation

Welcome to the assignment for candidates applying for a frontend position at Falcon.io.
In the following sections you can read up on the requirements and setup of the assignment.

Requirements

You're tasked with creating a prototype of the classic game 4-in-a-row. The game is a turn-based puzzle for two players, and the rules are simple:

  • The game board is made up of a 7x6 grid.
  • Whoever is first to get four pieces in a horizontal, vertical or diagonal line on the game board wins the game.
If you want to try out a version of the game to better understand the rules, a version can be found here.

We expect the submission to be handed in within 10-20 days.
If you, for any reason, estimate that you cannot finish the assignment within the timeframe please be communicative about an ETA.

If you navigate to http://localhost:8080 (or click "Back to Game" in the left-hand menu) you'll see a page with static markup demoing the initial layout of the game.
Your task is to turn this page into a Single Page Application hosting the game and the requirements hereof.
The index.html file is found in the /src folder. This folder is also your working directory where you'll implement the game.

Acceptance criteria

Below you find the business requirements for the game.
Note that there are requirements both towards the gameplay and the implementation it self.

  1. A player of the game must be able to:
    1. Start a game
    2. Join a game
    3. Win or lose a game
    4. (Optional) Leave/surrender a game
  2. The gameplay must work in realtime across two browser sessions (using the Websocket API layer provided in the boilerplate).
  3. Only two players can join a single game.
  4. Multiple game sessions can be running at the same time on the same Websocket API instance.
  5. The game is implemented using TypeScript. (Custom type definitions are encouraged, but not required).
  6. The game must, as a minimum, work in the latest version of Chrome or Firefox.
  7. (Optional) Handle client disconnects (e.g. auto-reconnect to a game if client gets disconnected unexpectectly).
  8. (Optional) Enable PWA (Progressive Web App) features, like background sync, offline support and push notifications.
  9. (Optional) Extend the build pipeline with a Webpack config that outputs production ready code.

The criteria marked as optional are, as the name suggest, not critical for the evaluation, and are not required to be part of your final submission.
Only if you find the extra time or feel like taking on the extra challenge, should you implement them.
They can, however, be talking points during a potential technical interview.
E.g.: How would you ensure that a distribution is production ready? or How would you handle flaky internet connections?.

Technical requirements

At Falcon.io we use Angular as the main platform to build applications, however you're free to use any framework and libraries to implement the game, or even go vanilla.
The important thing is that you feel comfortable in solving the core tasks and can defend your choices and patterns in a potential technical interview.

Regardless of your framework and library choices, there are certain areas that we expect you to address in your submission, since these will be considered doing the evaluations:

  1. You display modern best practices and good code hygiene.
  2. You stay (as much as possible) within the boilerplate setup. Any deviations you can defend with objective arguments.
  3. You follow Git practices and your commit history tells the story of how you iterated on the task.
  4. The game meets accessibility standards.
  5. The game is scalable, maintainable and pragmatic in its structure. Both on code and file level.
  6. The game is built for testability, demonstrated by actual tests.
  7. Complicated or non-obvious choices in the code must be supported by code-level comments.
  8. The README.md must contain a brief (a few lines) explanation of how you solved the task, as well as objective arguments for potential changes to the boilerplate.
Read more about how we evaluate your submission here.

Boilerplate

To get you started and to help you focus on the core tasks, we've provided you with a boilerplate setup.
The setup consist of:

  • Webpack development setup
  • Mocha test runner for unit tests
  • NPM scripts for bootstrapping
  • Websocket server with a minimal API
  • Websocket client with a minimal API
  • UI library
  • General HTML boilerplate
  • Premade game assets

Get started

The /src folder is your working directory. Here you'll find 4 index files, one for HTML, one for Typescript, one for SCSS(CSS) and one to aggregate unit tests.

With the exception of the webpack configuration the files supporting the boilerplate are found in the /lib folder.
Currently no framework or similar is preinstalled (with the exception of the ones supporting the Websocket API)
Note that if you wish to use the CLI tools of a given framework, e.g. ng new for Angular or create-react-app for React, make sure that the working directory remains /src, and that you document any new build pipeline in README.md.

The Webpack config is configured to transpile, bundle and serve up the application through a local development server. To start the development server run the following command (this will also serve this page): npm start

Websocket API

As mentioned in the requirements section the game must work in realtime across two browser sessions. To help you accomplish this we're providing a pre-defined Websocket setup. The setup consist of two layers:

  • A channel based Websocket server.
  • A client-side connector API based on observables (using RxJS).

To boot up the Websocket server run the following command: npm run ws_api

In index.ts you'll find a demo script with code examples of how the client connector works. Make sure to remove this script prior to submission.

Note that the Websocket API only implements the absolute minimum requirements for the realtime logic, and is only meant to provide a basic setup for you to solve the tasks.
You're free to change and/or extend the logic of the Websocket server and the client-side connector. Should you decide to do so, we expect that you briefly describe the changes in your documentation (in README.md) with objective arguments.

If you face issues when booting up the development server or API, make sure that you're running the LTS version of NodeJS, like v8.10.0

Writing tests

The boilerplate comes with a mocha test runner using chai for assertion and sinon for stubbing/mocking.

  • Mocha
  • Chai
  • Sinon

By running npm test all *.spec.ts will be picked up and run in sequence.

Example of tests format:

describe('example', () => {
   it('is true', () => {
    expect(true).to.be.true;
  });
});

If you decide to use a CLI tool to manage your framework of choice, e.g. ng for Angular, you're welcome to utilise the build-in test setup.
Just make sure to highlight this in README.md.

UI library

To support the overall visual aspect and layout, the boilerplate comes with Material Design Lite CSS framework.
You can find the documentation of Material Design Lite here.
There are framework bindings that you're free to use, although it is not required:

  • Angular
  • React

However, be aware, that Material Design Lite is also used for this documentation, so make sure to support both UI libraries to avoid a broken documentation page.

How to submit?

You submit your result via a Git repo. Preferably via GitHub.

Make sure that you have a Git history with meaningful commits that tells the story of how you solved the task.
A single init commit (or similar) contaning the entire solution is highly discouraged and will prevent us from doing proper evaluation.

If you've added build steps, additional NPM scripts or other steps that are required to make your implementation work, make sure that you've highlighted and described these in README.md

Make sure to fix the versions of any node_modules. This will ensure that the runtime environments will match.

  • BAD "some-package": "^0.3.1"
  • GOOD "some-package": "0.3.1"
Alternatively, you can create a Docker image holding the runtime environment.

IMPORTANT: A static Zip/RAR file containing the solution is not considered a valid submission and will be disregarded.

Once you're ready to submit send link to the remote Git repo to frontend-assignment@falcon.io

After submission

After you've submitted your result it will be evaluated by two or more members of the frontend team at Falcon.io.
You'll typically hear back from us within a week unless communicated otherwise.

If we conclude that your submission displays the technical skill level that we're looking for, we will invite you to a technical interview. Either onsite or remotely (via Hangout or similar) depending on your location and availability.
At the interview we will use your submission as the base of the conversation. We will talk about how you solved it, why you made the decisions you made and how you would improve it if you should refactor it.
If the optional requirements are not part of your submission, we will also be talking about these on a high level, i.e. how would you solve them, what are the pitfalls and is your current implementation flexible and scalable enough to accommondate new features?

How do we evaluate?

When we evaluate your submission there are some main areas that we will weight in:

  • Does your Git history reveal your workflow?
    When taking a first look at your Git repo we assess how you've iterated on the task.
    We do this by glancing through your Git commits. This will give us an idea of how you've priorised the sub-task and how you've iterated on those.
    E.g. how are business critical vs. non-critical implementations balanced?
    Therefore make sure that your commit messages provide specific context, e.g.:
    BAD some fixes
    GOOD fix(game-logic): fix issues with the winner calculation

    The Angular Commit Message Guidelines might prove useful for this.

  • Does your solution work out-of-the-box?
    Next, we read through your README.md and follow your instructions.
    Did you add custom steps? Do they work? Or, does it requires manual debugging to get up and running?

  • To what degree are the accepteance criteria met?
    Once everything is up and running, dev server, API etc., we open your solution in a browser and start playing the game.
    Does it work? Are all the requirements met? If not, we ask "why" and seek for answers in README.md or code-level comments.

  • How does it work?
    Provided that all of the acceptance criteria are up to standard, we start looking into how you implemented the solution.
    I.e. How did you decide to structure the application? How is it architectured? How do you manage state in the application? How scalable is your application?
    How performant is it? Which coding patterns and constructs do you utilise?
    Where and how do you handle edge cases? What about error handling? Is it accessible?

  • How's the sense of quality?
    While assessing the inner workings we also take a step back and look at the overall solution.
    How is it crafted? Does it feel solid, yet simple, in its execution? Is it built using common sense and pragmatic thinking?
    Is it well-tested? Is a good understanding of UX and application patterns applied?
    Is there a coherent idea submerging from the code and structure? What's the level of code smells?
    Are linting rules strictly followed?

Equally there are areas that will NOT weight heavily in our evaluation:

  • Look and feel

    We do not evaluate the overall visual look and feel.
    We expect that your main focus is spend on developing and fine-tuning application logic, which is also why we provided a UI library and pre-defined game assests.
    We do, however, expect that you show an understanding of common UX patterns and best practices, and can make good use of these when making UI/UX decisions.

  • Timeframe

    If nothing else is agreed, we expect the submission to be handed in within 10-20 days. However, we do not evaluate you on the time you spent on the assignment.
    I.e. we do not count the days from when you received the assignment till you submit, neither do we consider time intervals between your Git commits.

Enjoy!
